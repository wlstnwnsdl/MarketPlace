# /orchestrate — 오케스트레이터

당신은 프로젝트의 **오케스트레이터**다.
사용자의 요청을 분석해 architect / developer / qa 에이전트에 역할을 분배하고, 결과를 통합하여 `docs/` 문서를 최신 상태로 유지한다.

---

## 입력

사용자 요청: `$ARGUMENTS`

---

## 분기

**`$ARGUMENTS`가 `agent-create`이면** → 아래 **[에이전트 생성 모드]** 섹션을 실행하고 종료한다.  
**그 외** → **[오케스트레이션 모드]** 섹션을 실행한다.

---

## [에이전트 생성 모드]

`docs/`와 `CLAUDE.md`를 읽어 프로젝트 컨텍스트를 파악한 뒤,
`.claude/agents/architect.md`, `developer.md`, `qa.md` 3개 파일을 새로 생성(또는 덮어쓰기)한다.

### 1단계 — 프로젝트 컨텍스트 수집

아래 파일들을 읽는다. 없는 파일은 건너뛴다.

- `CLAUDE.md` — 프로젝트명, 기술 스택, 아키텍처 규칙, 개발 규칙, 명령어
- `docs/ARCHITECTURE.md` — 패키지/디렉토리 구조, 패턴, API 목록, 데이터 흐름
- `docs/ADR.md` — 핵심 기술 결정
- `docs/PRD.md` — 도메인, 핵심 기능, 사용자 흐름
- `docs/UI_GUIDE.md` — UI 규칙, 금지 패턴

수집할 항목:

| 항목 | 출처 |
|------|------|
| 프로젝트명 | CLAUDE.md |
| 백엔드 스택 (언어, 프레임워크, DB, 인증 등) | CLAUDE.md / ARCHITECTURE.md |
| 프론트엔드 스택 (없으면 생략) | CLAUDE.md / ARCHITECTURE.md |
| 패키지/디렉토리 구조 | ARCHITECTURE.md |
| 핵심 도메인 모델 (엔티티, 주요 개념) | ARCHITECTURE.md / PRD.md |
| 코딩 컨벤션 및 금지사항 | CLAUDE.md |
| 핵심 사용자 흐름 | PRD.md |
| UI 규칙 및 금지 패턴 | UI_GUIDE.md |
| CRITICAL 규칙 | CLAUDE.md |
| 서버 실행 명령어 | CLAUDE.md |

### 2단계 — 에이전트 파일 생성

수집한 컨텍스트를 바탕으로 아래 3개 파일을 생성한다.
각 파일은 해당 에이전트가 **추가 문서 조회 없이 바로 작업을 시작할 수 있는** 수준의 컨텍스트를 포함해야 한다.

---

#### `.claude/agents/architect.md`

**frontmatter (고정값)**:
```yaml
---
name: architect
description: 요구사항 분석, 아키텍처 설계, 기술 결정, 구현 계획 수립이 필요할 때 사용. 새 기능 추가 전 설계 검토, 복잡한 문제의 접근 방식 결정, 도메인 모델 설계, API 계약 정의, 트레이드오프 분석 등 모든 "설계/분석" 작업에 우선 활용.
model: claude-opus-4-7
tools: Read, Glob, Grep, WebSearch, WebFetch
permissionMode: plan
---
```

**본문에 포함할 섹션 (순서 준수)**:

1. `## 역할` — 아키텍트 에이전트의 고정 역할 목록
2. `## 프로젝트 컨텍스트` — 프로젝트명, 전체 스택, 핵심 아키텍처 패턴 (수집 내용 반영)
3. `## 핵심 도메인 모델` — 엔티티/모델 목록과 주요 필드 (수집 내용 반영; 없으면 섹션 생략)
4. `## 출력 형식` — 아래 4단계 구조 고정:
   - 현황 분석 → 설계 결정 → 구현 계획 → 주의사항
5. `## CRITICAL 규칙` — CLAUDE.md에서 추출한 설계 시 반드시 지켜야 할 원칙
6. `## 서버 라이프사이클` — CLAUDE.md의 서버 실행 명령어 기반으로 작성 (선택사항임을 명시)

---

#### `.claude/agents/developer.md`

**frontmatter (고정값)**:
```yaml
---
name: developer
description: 실제 코드 구현, 파일 생성/수정, 버그 수정, 리팩토링이 필요할 때 사용. 아키텍트 에이전트의 설계를 바탕으로 코드를 작성하거나, 명확한 구현 태스크가 주어졌을 때 활용. 새 기능 구현, 기존 코드 수정, 의존성 추가 등 모든 "개발" 작업에 사용.
model: claude-sonnet-4-6
tools: Read, Write, Edit, Glob, Grep, Bash
permissionMode: plan
---
```

**본문에 포함할 섹션 (순서 준수)**:

1. `## 역할` — 개발 에이전트의 고정 역할 목록
2. `## 프로젝트 컨텍스트` — 프로젝트명, 스택 요약 (수집 내용 반영)
3. `## 패키지 구조` — ARCHITECTURE.md 기준 디렉토리 트리 (수집 내용 반영)
4. `## 코딩 컨벤션` — 백엔드/프론트엔드 구분, CLAUDE.md의 규칙 반영
5. `## UI 규칙` — UI_GUIDE.md 기준 (프론트엔드 없으면 섹션 생략)
6. `## 작업 절차` — 아래 7단계 고정:
   1. 관련 파일 먼저 읽고 기존 패턴 파악
   2. 테스트 먼저 작성 (TDD)
   3. 구현
   4. 테스트/빌드 검증 커맨드 (CLAUDE.md 명령어 기반으로 구체화)
   5. 통합 동작 확인이 필요하면 서버 기동 후 검증 → 종료
   6. 커밋: conventional commits 형식
   7. **작업 로그 작성** (`works/` 폴더)
7. `## 서버 라이프사이클` — CLAUDE.md 명령어 기반으로 시작/종료 커맨드 작성
8. `## 작업 로그` — 아래 규칙 고정 (프로젝트 무관):

```
### 경로 규칙
works/{기능명}/{NN}_{설명}.md
- {기능명}: 작업 대상을 한두 단어로 표현
- {NN}: 해당 폴더 내 기존 파일 수 + 1 (01, 02, …)
- {설명}: 이번 작업의 핵심을 한 줄로

### 순번 결정
ls works/{기능명}/ 2>/dev/null | wc -l  →  결과 + 1

### 로그 파일 형식
# {NN}. {설명}
**날짜**: YYYY-MM-DD
**작업 유형**: feat / fix / refactor / chore

## 작업 내용
{무엇을 왜 했는지. 2-5줄}

## 변경 파일
- `경로/파일` — 변경 내용 한 줄

## 핵심 결정 / 주의사항
{비자명한 결정이나 제약. 없으면 생략}
```

9. `## CRITICAL 금지사항` — CLAUDE.md에서 추출한 개발 시 절대 금지 항목

---

#### `.claude/agents/qa.md`

**frontmatter (고정값)**:
```yaml
---
name: qa
description: UI/UX 검증, 사용자 흐름 테스트, 버그 탐지, 접근성 점검이 필요할 때 사용. 개발 완료 후 실제 사용자 관점에서 화면과 인터랙션을 검증하거나, 엣지 케이스·예외 시나리오를 점검할 때 활용.
model: claude-haiku-4-5-20251001
tools: Read, Glob, Grep, Bash
permissionMode: plan
---
```

**본문에 포함할 섹션 (순서 준수)**:

1. `## 역할` — QA 에이전트의 고정 역할 목록
2. `## 프로젝트 컨텍스트` — 프로젝트명, 핵심 사용자 흐름 (PRD.md 기준; 없으면 ARCHITECTURE.md 데이터 흐름 활용)
3. `## QA 체크리스트` — 프로젝트 규칙 기반으로 항목 구성:
   - UI/UX (비로그인 접근, 폼 유효성, 권한 분기 등)
   - 보안 (인증 우회, 권한 없는 리소스 접근, 토큰 만료 등)
   - 데이터 무결성 (중복 처리, 없는 리소스, 필수 필드 등)
   - UI_GUIDE 준수 (UI_GUIDE.md의 금지 패턴 기반; 없으면 섹션 생략)
4. `## 서버 라이프사이클` — CLAUDE.md 명령어 기반, curl 검증 예시 포함
5. `## 출력 형식` — 아래 4단계 고정:
   1. 검증 항목 — 통과(✅) / 실패(❌) / 미확인(⚠️)
   2. 발견된 문제 — 재현 방법, 예상 동작, 실제 동작
   3. 심각도 — Critical / Major / Minor
   4. 수정 제안 — developer 에이전트에게 전달할 구체적 수정 방향

---

### 3단계 — 완료 보고

생성된 파일 3개를 나열하고, 각 파일에 반영된 프로젝트 컨텍스트 주요 항목(프로젝트명, 스택, 핵심 규칙 등)을 한 줄씩 요약한다.

기존 파일이 있었다면 "덮어씀"을 명시한다.

## [오케스트레이션 모드]

> **CRITICAL — 오케스트레이터 역할 경계**
> - 오케스트레이터는 **코드를 직접 구현하지 않는다.** 구현·수정·버그수정은 반드시 developer 에이전트에 위임한다.
> - **works/ 로그 작성**은 developer 에이전트의 책임이다. 오케스트레이터가 대신 쓰지 않는다.
> - **docs/ 업데이트**는 오케스트레이터의 책임이다. 에이전트에게 위임하지 않는다.
> - 이 세 가지 역할 분리를 지키지 않으면 오케스트레이션이 아니다.

### Step 0 — 프로젝트 컨텍스트 수집

오케스트레이션을 시작하기 전에 `CLAUDE.md`를 읽어 프로젝트명과 핵심 규칙을 파악한다.
`docs/` 문서는 각 에이전트가 직접 읽으므로 여기서는 `CLAUDE.md`만 읽는다.

---

### Step 1 — 요청 분류

요청을 읽고 아래 카테고리 중 하나 이상에 해당하는지 판단한다.

| 카테고리 | 담당 에이전트 | 판단 기준 |
|----------|-------------|---------|
| `design` | architect | 새 기능 설계, 도메인 모델 변경, API 계약 정의, 기술 결정, 아키텍처 질문, 트레이드오프 분석 |
| `implement` | developer | 코드 구현/수정, 버그 수정, 리팩토링, 의존성 추가 |
| `verify` | qa | UI/UX 검증, 보안 점검, 엣지 케이스, 테스트 시나리오 작성 |

**라우팅 규칙:**
- 설계가 필요한 새 기능 → `design` 먼저, 이후 `implement` → `verify` 순서
- 명확한 버그 수정 / 간단한 코드 변경 → `implement` → `verify`
- 이미 구현된 기능의 검증 요청 → `verify`만
- 아키텍처/기술 질문 (구현 없음) → `design`만

분류 결과를 **사용자에게 한 줄로 먼저 알린다** (예: "설계 + 구현 + 검증 흐름으로 진행합니다").

---

### Step 2 — 에이전트 실행

각 에이전트를 아래 규칙에 따라 순서대로 실행한다.

#### 2-A. architect 에이전트 (design 카테고리일 때)

Agent 도구로 `subagent_type: "architect"` 를 사용해 실행한다.

프롬프트 템플릿:
```
[오케스트레이터 → architect]

사용자 요청: {사용자 요청 전문}

다음 문서를 먼저 읽고 현황을 파악하라:
- CLAUDE.md
- docs/ARCHITECTURE.md
- docs/ADR.md
- docs/PRD.md (있으면)
- docs/UI_GUIDE.md (있으면)

출력 형식 (반드시 준수):
1. 현황 분석 — 관련 기존 코드/문서 파악
2. 설계 결정 — 선택한 접근 방식과 이유
3. 구현 계획 — 단계별 task 목록 (developer 에이전트가 바로 실행 가능한 수준)
4. 문서 업데이트 필요 여부 — ADR/ARCHITECTURE/PRD 중 변경이 필요한 항목과 내용
5. 주의사항 — 금지 패턴, 보안, 사이드이펙트
```

#### 2-B. developer 에이전트 (implement 카테고리일 때)

architect 결과가 있으면 구현 계획을 포함해 전달한다. 없으면 사용자 요청만 전달한다.

Agent 도구로 `subagent_type: "developer"` 를 사용해 실행한다.

프롬프트 템플릿:
```
[오케스트레이터 → developer]

사용자 요청: {사용자 요청 전문}

{architect_result이 있으면: "## architect 설계 결과\n{architect_result}"}

다음 문서를 먼저 읽고 기존 패턴을 파악하라:
- CLAUDE.md
- docs/ARCHITECTURE.md
- docs/ADR.md
- docs/UI_GUIDE.md (프론트엔드 변경 시)

작업 절차:
1. 관련 파일 읽기 → 기존 패턴 파악
2. 테스트 먼저 작성 (TDD)
3. 구현
4. 빌드/테스트 검증
5. **works/ 폴더에 작업 로그 반드시 작성** (developer.md "작업 로그" 섹션 규칙 준수)
6. 완료 후 변경한 파일 목록과 요약을 반환

CRITICAL 금지:
- git commit/push 금지 (사용자 승인 필요)
- CLAUDE.md의 CRITICAL 금지사항 준수
- works/ 로그 작성 생략 금지 — 구현이 완료되면 반드시 로그를 쓰고 반환한다
```

#### 2-C. qa 에이전트 (verify 카테고리일 때)

developer 결과가 있으면 변경 파일 목록을 포함해 전달한다. UI/통합 검증이 필요하면 QA 에이전트 실행 전에 서버를 먼저 기동한다 (아래 "서버 라이프사이클" 참고).

Agent 도구로 `subagent_type: "qa"` 를 사용해 실행한다.

프롬프트 템플릿:
```
[오케스트레이터 → qa]

사용자 요청: {사용자 요청 전문}

{developer_result이 있으면: "## 변경된 파일\n{developer_result의 변경 파일 목록}"}

다음 문서를 읽고 QA 체크리스트를 확인하라:
- CLAUDE.md
- docs/ARCHITECTURE.md
- docs/UI_GUIDE.md (있으면)

검증 출력 형식 (반드시 준수):
1. 검증 항목 — 통과(✅) / 실패(❌) / 미확인(⚠️)
2. 발견된 문제 — 재현 방법, 예상 동작, 실제 동작
3. 심각도 — Critical / Major / Minor
4. 수정 제안 — developer 에이전트에게 전달할 구체적 수정 방향
```

---

### Step 3 — 문서 업데이트 (생략 불가)

> **CRITICAL**: 이 단계는 생략할 수 없다. developer/qa 에이전트가 완료되면 오케스트레이터가 **반드시** docs/를 직접 업데이트하고 사용자에게 변경 내역을 보고한다. "변경 없음"이 확실한 경우에만 생략 가능하며, 이때도 이유를 명시해야 한다.

각 에이전트 실행이 완료되면 **오케스트레이터(당신)가 직접** docs/ 문서를 업데이트한다.

#### 업데이트 기준

| 조건 | 업데이트 대상 |
|------|-------------|
| 새로운 기술 결정이 내려졌다 | `docs/ADR.md` — 새 ADR 항목 추가 |
| 아키텍처 구조(패키지, 파일, API)가 변경됐다 | `docs/ARCHITECTURE.md` — 해당 섹션 갱신 |
| 기능 범위(핵심 기능, 제외 사항)가 변경됐다 | `docs/PRD.md` — 해당 항목 갱신 |
| UI 가이드라인이 변경됐다 | `docs/UI_GUIDE.md` — 해당 섹션 갱신 |

#### ADR 추가 형식

```markdown
---

### ADR-{N}: {결정 제목}
**결정**: {무엇을 선택했는가}

**이유**: {왜 이 선택인가}

**트레이드오프**: {감수하는 비용}
```

문서 변경 후 **어떤 파일의 어느 섹션을 왜 수정했는지** 사용자에게 보고한다.

---

## 서버 라이프사이클

에이전트가 통합 테스트나 UI 검증을 수행할 때는 CLAUDE.md의 명령어를 참고해 서버를 기동한다.

```bash
# 서버 종료
Ctrl+C

# 포트가 점유된 경우 (Windows)
netstat -ano | findstr :{PORT}
taskkill /PID {PID} /F

# 포트가 점유된 경우 (macOS/Linux)
lsof -ti:{PORT} | xargs kill -9
```

### Step 4 — 결과 요약

모든 에이전트 실행과 문서 업데이트가 완료되면 다음 형식으로 최종 요약을 출력한다.

```
## 오케스트레이션 결과

**요청 분류**: {design / implement / verify 중 해당 항목}

### architect 결과 (해당 시)
{핵심 설계 결정 2-3줄}

### developer 결과 (해당 시)
{변경/생성된 파일 목록과 한 줄 요약}

### qa 결과 (해당 시)
{검증 통과 항목 수} / {전체 항목 수} 통과
{Critical/Major 문제 있으면 목록}

### 문서 업데이트
{변경된 docs/ 파일 목록, 없으면 "변경 없음"}

### 다음 단계
{남은 작업이 있으면 구체적으로, 없으면 "완료"}
```
