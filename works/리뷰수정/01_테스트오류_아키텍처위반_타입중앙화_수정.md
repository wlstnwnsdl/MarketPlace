# 01. 테스트 컴파일 오류 수정, 아키텍처 위반 제거, 타입 중앙화

**날짜**: 2026-04-22  
**작업 유형**: fix / refactor

## 작업 내용

QA 리뷰에서 발견된 4가지 우선순위 이슈를 수정했다.

- [P1] `PromptRequest`에 `PromptStatus status` 필드가 추가되면서 테스트 3곳(PromptControllerTest, PromptServiceTest, PromptRepositoryTest)에서 컴파일 오류가 발생했다. 각 호출 지점에 status 인자를 추가하고, PromptServiceTest의 setUp에서 기본 prompt를 PUBLIC 상태로 명시해 findPromptDetail 로직(PUBLIC이 아닌 프롬프트는 404)과 일치시켰다.
- [P2] `PurchaseController`가 `PromptRepository`와 `UserRepository`를 직접 주입받아 CLAUDE.md 아키텍처 규칙을 위반하고 있었다. `UserService`를 신규 생성하고, `PromptService`에 `getMyPrompts()` 메서드를 추가했다. `UserNotFoundException`도 도메인 예외 패턴에 맞게 신규 생성하고 `GlobalExceptionHandler`에 등록했다.
- [P3] `GET /api/users/me`, `GET /api/users/me/prompts` 엔드포인트에 대한 컨트롤러 통합 테스트(`PurchaseControllerTest.java`)를 새로 작성했다.
- [P4] `frontend/src/api/user.ts`에 중복 정의되어 있던 `UserProfile` 인터페이스를 `frontend/src/types/index.ts`로 이동했다. 사용하는 파일(Header.tsx, HomePage.tsx)의 import 경로도 함께 수정했다.

## 핵심 결정 / 주의사항

- PromptRepositoryTest의 기존 3개 테스트는 status=null을 전달해 전체 조회하도록 수정했다. 테스트 데이터는 PENDING 상태이므로 PUBLIC 필터를 적용하면 0건이 나온다. 향후 findWithFilters를 사용할 때 반드시 status를 명시해야 한다.
- PromptServiceTest setUp에서 prompt를 PUBLIC 상태로 설정한 것은 `findPromptDetail` 비즈니스 로직(소유자가 아닌 경우 PUBLIC이 아니면 404)과의 정합성을 위한 결정이다.
- `UserService`는 현재 `getById()` 하나만 제공한다. OAuth2UserService는 별도 레이어(config/)에서 User를 직접 저장하므로 UserService와 충돌하지 않는다.
